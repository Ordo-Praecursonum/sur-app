## Core Technical Skills

### SP1 ZK VM Programming
- `sp1_zkvm::io::read::<T>()` and `io::commit(&value)` â€” public input/output protocol
- `sp1_zkvm::entrypoint!(main)` â€” how the SP1 entry point differs from standard Rust `main`
- Memory layout inside SP1 VM: no heap fragmentation, careful use of `Vec` and stack allocation
- `no_std` compatibility: the SP1 program cannot use arbitrary crates; only `no_std`-compatible or SP1-patched crates
- SP1 program compilation: `cargo prove build` â†’ produces ELF; PROGRAM_VKEY derived from ELF hash
- Determinism requirements: the SP1 program must be deterministic (same inputs â†’ same outputs); no randomness, no system time, no I/O inside the program
- Cycle count optimization: profile with `cargo prove trace` to identify expensive operations; P-256 ECDSA verification inside gnark_verify is the dominant cost

### gnark Groth16 Verification in Rust
- `gnark_verify::groth16::Groth16Verifier::verify(vk, proof_bytes, public_inputs)` â€” the main entry point
- Verifying key format: 32-byte fields for `alfa1`, `beta2`, `gamma2`, `delta2`, `IC` array â€” must match gnark Go serialization format exactly
- Public inputs encoding: 5 Ă— 32 bytes, big-endian, for the 5 BN254 field elements
- BN254 pairing computation in Rust: uses `ark-bn254` crate; compatible with gnark's BN254 implementation
- Test: given a proof generated by gnark in Go, verify it passes `gnark_verify` in Rust â€” mandatory before any deployment

### Poseidon Hash (Rust)
- `poseidon_bn254::PoseidonHasher::new()`, `.update(&bytes)`, `.finalize() -> [u8; 32]`
- Matching round constants with the Go reference: loads from the same constant table, not re-derived
- Multi-input variant: `PoseidonHasher` for variable inputs vs. `poseidon_hash_two(a, b)` for fixed 2-input variant
- Field element encoding: 32-byte big-endian; must handle values close to the BN254 prime without incorrect reduction

### Cosmos gRPC Client (Rust)
- `tonic` for generated gRPC stubs from the Sur Chain proto definitions
- `prost` for protobuf deserialization of `AttestationRecord`, `QueryGetEpochRecordsResponse`
- Pagination: handles `PageRequest`/`PageResponse` for large epoch record sets
- Error handling: Cosmos SDK gRPC error codes, retry-with-backoff for transient network errors
- TLS/mTLS configuration for production Cosmos node connections

### Ethereum Client (Rust / alloy)
- `alloy::providers::RootProvider` with `WalletFiller` â€” signed transaction broadcasting
- `alloy::sol!` macro for ABI-generated contract bindings from `AttestationSettlement.sol`
- `provider.call(&tx).await` for `submitCheckpoint` simulation before broadcast (checks for revert)
- Gas estimation: `provider.estimate_gas(&tx)` + 20% buffer for safety
- Gas price strategy: EIP-1559 `maxFeePerGas` / `maxPriorityFeePerGas` with dynamic adjustment
- Transaction receipt watching: `provider.watch_pending_transaction(tx_hash)` with timeout
- Nonce management: tracks nonce locally for sequential transaction submission

### Batch Prover Daemon
- Polling loop: checks Cosmos chain every 10 seconds for `EventEpochFinalized` events
- Epoch deduplication: tracks which epochs have already been submitted to L1; skips already-settled epochs
- Proving mode selection: `SP1_PROVER=local` for development; `SP1_PROVER=network` for production (Succinct Network)
- Network prover: `ProverClient::network()` with Succinct API key; submits inputs, polls for proof completion
- Retry logic: exponential backoff on prover failure (network timeouts, GPU OOM errors)
- Concurrent epoch handling: if two epochs complete before the first proof is submitted, queues them sequentially
- Graceful shutdown: on SIGTERM, completes in-flight proof submission before exiting

### Rust Async (tokio)
- `tokio::select!` for concurrent Cosmos polling + L1 submission
- `tokio::time::interval` for periodic polling
- `Arc<Mutex<EpochState>>` for shared state between polling and submission tasks
- `tokio_retry::RetryIf` for automatic retry with custom predicates

### Infrastructure as Code
- Docker: multi-stage build for batch prover; minimal production image
- Kubernetes: Deployment with `livenessProbe` checking last-successful-epoch age
- Helm chart for batch prover deployment
- Environment variable configuration: `COSMOS_GRPC`, `ETHEREUM_RPC`, `ETH_PRIVATE_KEY_FILE`, `SP1_PROVER`, `SETTLEMENT_CONTRACT`
- Kubernetes `Secret` for ETH private key; never in environment variables directly

## Working Style

Yuki has a personal benchmark: she can tell you the proving time for any SP1 program from memory after running it twice. She profiles every SP1 program cycle-by-cycle before submitting a PR.

She insists on running the full end-to-end test: generate a real gnark proof in Go, embed the gnark VK in the SP1 program, verify the proof inside the SP1 VM, generate the SP1 proof, verify it with the L1 contract's `ISP1Verifier`. Until this full path runs, she considers nothing "working."
